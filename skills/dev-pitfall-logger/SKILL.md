# Pitfall Logger - 开发踩坑记录器

> **核心原则**: 改规则，不改产出。遇到问题时先修 SKILL.md，再处理数据。

`★ Insight ─────────────────────────────────────`
**为什么重要**: 开发者 20-30% 的时间在重复解决已解决过的问题。这个 Skill 将这些浪费转化为可累积的知识资产。关键不只是记录"怎么修"，而是"为什么"和"如何避免"。
`─────────────────────────────────────────────────`

## 🎯 第一步：决策树（最优先执行）

```
用户请求 → 判断类型 → 执行对应任务

┌─────────────────────────────────────────────┐
│ 用户说了什么？                               │
├─────────────────────────────────────────────┤
│ "上次/之前/查一下/怎么解决的" → Task 1: 搜索 │
│ "记录/记一下/这个坑"           → Task 2: 创建 │
│ "总结一下今天/这个项目"     → 总结后主动询问是否记录 │
│ [解决问题后，满足触发条件]     → Task 4: 主动记录 │
│ "回顾/总结/本月/本周"         → Task 5: 定期回顾 │
└─────────────────────────────────────────────┘
```

**执行顺序（必须遵守）**:
1. 先执行决策树，确定任务类型
2. 再加载对应的详细工作流（如果有）
3. 执行任务

## 🔑 核心原则（不可违反）

### 1. 治本不治标（最重要）

**遇到问题时，修改规则而不是修改产出物**

| 场景 | ❌ 错误做法 | ✅ 正确做法 |
|------|-----------|-----------|
| 搜索结果不准确 | 手动筛选结果 | 检查搜索关键词策略，优化规则 |
| 记录格式混乱 | 手动调整格式 | 检查模板，补充格式规则 |
| 主动记录太频繁 | 跳过某些记录 | 调整触发条件阈值 |
| 索引更新失败 | 手动补索引 | 检查索引生成逻辑 |

**判断标准**: 问题是"规则不够完善"还是"执行出错"？如果是前者，改规则。

### 2. 量化判断标准（可执行）

所有"是否做X"的判断必须用可量化的标准：

| 判断 | 量化标准 | 不接受 |
|------|---------|--------|
| 是否主动记录 | 解决时间 > 10分钟 OR 尝试次数 ≥ 2 | "看起来挺复杂" |
| 是否高严重程度 | 生产环境阻塞 OR 数据丢失风险 OR 修复时间 > 1小时 | "挺严重的" |
| 是否需要根因分析 | 严重程度 ≥ medium OR 属于"非直觉问题" | "有空再分析" |
| 是否值得预防 | 同类问题可能出现 2+ 次 | "以后再说" |

### 3. 单一数据源

**规则**: 踩坑记录只存储在 `references/logs/YYYY/YYYY-MM.md`

索引文件（`by-category/*.md`, `by-project/*.md`）**只包含链接，不存储数据**。

**检查方法**: 如果发现索引文件包含完整内容，说明规则被违反了，需要修复。

### 4. 跨文件一致性检查

**当产出物出现问题时，第一步检查所有规则文件**：

```bash
# 检查是否有冲突定义
grep -r "关键词定义" references/
```

**如果同一定义在多处出现，必须统一到一处**。

## ⚡ 快速参考

### 分类标准
```
frontend  → UI组件、框架、样式、浏览器
backend   → API、服务端逻辑、中间件、认证
database  → 查询、迁移、连接、ORM
devops    → 部署、CI/CD、Docker、Kubernetes
tooling   → 构建工具、包管理器、IDE、git
other     → 项目管理、文档、流程
```

### 严重程度标准
```
critical → 生产宕机、数据丢失、安全漏洞
high     → 功能阻塞、需紧急修复
medium   → 有变通方案、影响体验
low      → 小问题、快速修复
```

### 文件路径
```
records/     → references/logs/YYYY/YYYY-MM.md
index/       → references/index.md
categories/  → references/logs/by-category/{category}.md
projects/    → references/logs/by-project/{project}.md
template/    → references/templates/pitfall_template.md
```

## 📋 详细工作流

### Task 1: 搜索记录
→ 详细规则: `references/workflows/search.md`

**快速判断**:
- 用户说"上次/之前/查一下/怎么解决的" → 执行搜索
- 提取关键词 → 优先查分类索引 → 再查全文
- 返回结果用 ⚡💡📌 格式总结

### Task 2: 创建记录
→ 详细规则: `references/workflows/create.md`

**快速判断**:
- 用户说"记录/记一下/这个坑" → 创建新记录
- 加载模板 → 收集信息 → 生成ID → 保存到月文件 → 更新索引

### Task 3: 组织索引
→ 详细规则: `references/workflows/index.md`

**快速判断**:
- 创建记录后自动执行
- 用户请求回顾时执行
- 只更新链接，不复制数据

### Task 4: 主动记录
→ 详细规则: `references/workflows/proactive.md`

**触发场景（任一满足即触发）**:

**场景1: 解决复杂问题后**
1. ✅ 解决时间 > 10分钟
2. ✅ 尝试次数 ≥ 2（不同方案）
3. ✅ 非"明显常识问题"
4. ✅ 非快速解决（< 2分钟）

**场景2: 项目总结/每日回顾后**
1. ✅ 用户要求"总结一下今天的工作"
2. ✅ 用户要求"总结一下这个项目的问题"
3. ✅ 完成阶段性工作回顾

**触发后自然提议**:
```
📌 这个问题花了些时间才解决，需要记录到踩坑库吗？
```

或

```
📌 今天的总结中有X个值得记录的问题，需要我帮你记录到踩坑库吗？
```

### Task 5: 定期回顾
→ 详细规则: `references/workflows/review.md`

**触发方式**:
- 用户主动询问
- 创建记录时轻量提醒（可选）
- 项目结束时

**输出格式**:
```
📊 本月踩坑回顾（共N条）
🔴 High: X条 | 🟡 Medium: Y条 | 🟢 Low: Z条

值得关注的模式：
1. XXX问题反复出现 → 建议：XXX
2. XXX浪费大量时间 → 建议：XXX
```

## 🎨 输出格式标准

### 智能摘要格式（搜索结果）
```markdown
⚡ 问题：[1句话概括]
💡 解决：[关键步骤]
📌 关键：[最重要1点]
```

### 记录ID格式
```
pit-YYYYMMDD-NNN
例: pit-20260224-001
```

### 索引链接格式
```markdown
- **[pit-20260224-001](../../2026/2026-02.md#pit-20260224-001):** 简短标题
  - Category: frontend | Severity: medium
```

## ⚠️ 常见错误与修复

| 错误现象 | 根因 | 修复规则 |
|---------|------|---------|
| 搜索结果为空 | 关键词太具体 | 添加模糊搜索策略 |
| 索引用数据重复 | 违反单一数据源 | 强制只存链接 |
| 主动记录太烦 | 触发阈值太低 | 提高时间/次数门槛 |
| 重复记录同一问题 | 缺少查重逻辑 | 创建前先搜索相似问题 |
| 记录格式混乱 | 模板不够具体 | 添加必填字段验证 |

## 📦 渐进式加载说明

本 SKILL.md 只包含核心决策和最常用的规则。详细工作流程拆分到独立文件：

```
references/workflows/
├── search.md   → 搜索详细步骤
├── create.md   → 创建详细步骤
├── index.md    → 索引详细步骤
├── proactive.md → 主动记录详细逻辑
└── review.md   → 回顾详细逻辑
```

**何时加载详细规则**:
- 确定任务类型后，按需加载对应文件
- 不要一次性加载所有文件（避免遗忘）
