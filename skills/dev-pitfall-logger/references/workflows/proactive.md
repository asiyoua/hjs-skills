# 主动记录 - 详细工作流

## 核心原则

**在正常项目对话中，自动检测值得记录的踩坑时刻，主动提议记录。**

**不主动原则**: 不要为了记录而打断对话流程，只在合适时机提议。

---

## 触发条件（必须全部满足）

### 1. 时间条件（满足任一）
- ✅ 解决一个问题花费 **> 10分钟**
- ✅ 尝试了 **2种以上不同方案**
- ✅ 用户表达 **"终于解决了"**

### 2. 问题条件（必须满足）
- ✅ 非"明显常识问题"（如 npm install 安装依赖）
- ✅ 非"快速解决的拼写/语法错误"
- ✅ 解决方案 **不是显而易见的**（需要查资料/尝试）

### 3. 复用条件（满足任一）
- ✅ 这个问题 **可能在其他场景再次出现**
- �这个问题 **涉及不常见的技术栈或配置**
- ✅ 这个问题 **包含有价值的调试过程**

---

## 不触发的情况（必须全部满足才跳过）

### 明显不需要记录
- ❌ 明显的拼写错误或语法错误
- ❌ 简单的配置修改（改个端口号）
- ❌ 众所周知的常识（如 git add 提交文件）
- ❌ 快速解决（< 2分钟）的问题

### 用户明确不需要
- ❌ 用户之前说过"不用记录这类问题"
- ❌ 用户在本次对话中明确表示"不用记录"

---

## 提议时机

### 最佳时机
**问题解决后的自然停顿点**：

```bash
# 场景1：完成一个功能后
[问题解决，代码运行成功]

Claude: 问题已经解决了！
      📌 这个问题花了些时间才解决，需要记录到踩坑库吗？

User: 好的

Claude: [开始创建记录]
```

```bash
# 场景2：尝试多个方案后
[尝试方案1，失败]
[尝试方案2，成功]

Claude: 方案2成功了！
      📌 这个坑值得记一下吗？方便以后遇到类似问题时快速查找。
```

### 时机判断标准

| 对话状态 | 是否提议 | 原因 |
|---------|---------|------|
| 问题刚解决，有停顿 | ✅ 是 | 自然间隙 |
| 用户还在测试 | ❌ 否 | 不要打断 |
| 还有其他任务 | ❌ 否 | 避免干扰 |
| 用户表达"终于解决了" | ✅ 是 | 用户认可问题有价值 |

---

## 提议方式

### 简洁自然（推荐）
```markdown
📌 这个问题花了些时间才解决，需要记录到踩坑库吗？
方便以后遇到类似问题时快速查找。
```

### 更简洁（快速对话中）
```markdown
📌 这个坑值得记一下吗？
```

### 附带预览（第一次记录时）
```markdown
💡 我会从对话中自动提取：
- 问题：XXX
- 解决：XXX
- 技术栈：XXX

只需确认分类和严重程度，其他自动补全。
要记录吗？
```

---

## 用户响应处理

### 响应类型及处理

| 用户说 | 处理方式 |
|--------|---------|
| "记一下"/"好的"/"需要"/"可以" | 立即创建记录 |
| "不用"/"算了"/"不用了" | 不再追问，继续对话 |
| "再看看"/"稍后" | 暂不记录，继续对话 |
| [没回应] | 默认记录，简短确认"已记录" |

### 自动提取信息

用户同意后，从当前对话中自动提取：

| 信息 | 提取方式 | 示例 |
|------|---------|------|
| 项目名称 | 工作目录或对话上下文 | `WeChatProjects/miniprogram-2` → miniprogram |
| 错误现象 | 用户描述的问题 | "npm install在公司网络失败" |
| 解决方案 | 刚实施的解决步骤 | "配置.npmrc使用公司镜像" |
| 技术栈 | 涉及的技术/工具 | [npm, proxy, corporate-network] |

**只需确认**:
- 分类（frontend/backend/database/devops/tooling/other）
- 严重程度（low/medium/high/critical）

---

## 信息补全策略

### 信息不足时的处理

```markdown
从对话中我已提取：
✅ 问题：XXX
✅ 解决：XXX
✅ 技术：[XXX]

还需要确认：
❓ 分类：这是哪类问题？
   [1] frontend  [2] backend  [3] database
   [4] devops    [5] tooling   [6] other

❓ 严重程度：
   [1] low（小问题，快速修复）
   [2] medium（有变通方案）
   [3] high（功能阻塞）
   [4] critical（生产宕机/数据丢失）
```

---

## 避免打扰原则

### 不打断工作流
- ❌ 不在用户还在调试时提议
- ❌ 不在用户快速尝试多个方案时频繁提议
- ✅ 在问题解决、有自然停顿时提议

### 一句话可以跳过
- 提议必须简洁，一句话能看完
- 用户说"不用"就停止，不再追问

### 不强制记录
- 用户可以随时拒绝
- 不会因为拒绝而影响后续对话

---

## 触发条件示例

### ✅ 应该触发

```
# 示例1：时间条件满足
[尝试方案A，15分钟后失败]
[尝试方案B，又花10分钟，成功]

Claude: 方案B成功了！
      📌 这个问题花了些时间才解决，需要记录到踩坑库吗？
```

```
# 示例2：多次尝试
[使用 Promise.all，失败]
[改用 Promise.allSettled，失败]
[最终发现是数据格式问题，修复后成功]

Claude: 终于解决了！
      📌 这个坑值得记一下吗？
```

```
# 示例3：用户表达"终于解决了"
User: 呼，终于解决了，这个问题搞了我半天

Claude: 📌 这个问题花了些时间，需要记录到踩坑库吗？
```

### ❌ 不应该触发

```
# 示例1：快速解决
User: 这个端口被占用了

Claude: 改成另一个端口试试

User: 好的，解决了

Claude: [不提议记录，因为<2分钟解决]
```

```
# 示例2：明显常识
User: npm install 报错

Claude: 运行 npm install 安装依赖

User: 好的

Claude: [不提议记录，因为这是常识操作]
```
